Section 01

What It Does

Cephlow is a certificate generation and delivery SaaS. Organizations use it to bulk-generate and send PDF certificates to students via email and WhatsApp. A user connects their Google account, picks a Google Sheet (participant list) and a Google Slides template (certificate design), then clicks Generate. The app fills the template for each row client-side, exports it as a PDF, uploads it to Cloudflare R2, and sends it via Gmail or WhatsApp Business API.

Recipients can text the WhatsApp number at any time to re-download their certificate. A Cloudflare Worker handles the bot and bridges customer support conversations to Telegram topics.

Key Pattern — OpenAPI Codegen

Section 05

Backend API — Hono on Cloudflare Workers

Edge-native Hono API backend running on Cloudflare Workers. It handles all database transactions directly on Cloudflare D1 with zero network latency overhead.

Route Architecture

apps/api-worker/src/index.ts — Route Groups
PUBLIC (no auth required):
  GET/POST /api/webhooks/whatsapp   → WA webhook verify + delivery status
  POST     /api/webhooks/cashfree   → Payment confirmation from Cashfree
  GET      /api/verify/:certId      → QR code certificate verification
  GET      /api/qr/:certId          → QR image (Google Slides embeds it)
  POST     /api/internal/report-*   → Server-to-server (shared-secret header)

AUTH REQUIRED (Supabase JWT):
  /api/batches/*          → Batch CRUD + generate + send
  /api/certificates/*     → Certificate records
  /api/sheets/*           → Google Sheets integration
  /api/slides/*           → Google Slides integration
  /api/wallet/*           → Credits and payment ledger
  /api/workspaces/*       → Multi-tenant workspace management

Section 06b

Object Storage — Cloudflare R2

Cloudflare R2 provides S3-compatible, zero-egress-fee object storage at the edge. It is used to store and serve the generated certificate PDF files securely and globally.

R2 Architecture & Access Control

01
Presigned Upload URLs
To prevent loading large PDF buffers in Worker memory, the Hono backend generates a presigned S3 PUT URL. The React frontend uploads the PDF directly from the client browser to R2.
02
Public Certificate Access
Certificates are stored under paths matching their unique public IDs. R2 buckets are bound to a custom domain with cache headers configured to allow instant verification loading.
03
Directory Structure for WhatsApp Bot
Files are structured with a phone-number prefix key structure. This enables the WhatsApp chatbot to query the storage and locate previous certificates belonging to a recipient rapidly.
Section 07

Certificate Generation Pipeline

▶
Client-Side Generation (Browser-Native)

The browser renders and copies PDF certificates client-side. This leverages the client's local CPU/RAM and uses their personal Google API quota (60 writes/minute) rather than exhausting a centralized service account quota.

01
User Configures Batch
Selects Google Sheet (participant data) + Google Slides template + column mappings.
02
Frontend Fetches Data
Browser pulls Sheets data via Google Sheets APIs and requests a Google access token.
03
Browser: Copy & Edit Slides
Browser calls Google Drive and Slides APIs to copy templates and replace placeholders (e.g. {{Name}}) for each participant.
04
Browser: Export to PDF
Calls Google Drive's PDF export API directly from the client.
05
Request R2 Presigned URL
Frontend requests a presigned S3 PUT URL from Hono backend.
06
Direct Upload to R2
Browser performs a direct PUT request upload to Cloudflare R2. PDF bytes never hit the Worker.
07
Update Metadata in D1
Frontend notifies Hono to record the generated certificate metadata in D1.
Section 08

WhatsApp Business API

Direct integration with the Meta WhatsApp Cloud API, bypassing third-party vendors. Requests go directly to Meta Graph API endpoints.

Template Formatting

Uses approved template variables (e.g. document_senderv3) to deliver the certificate as a header attachment alongside recipient and verification page details.

Delivery Tracking

Webhooks from Meta are received by the Worker and processed in-line. The status of delivery is atomically tracked (`sent` → `delivered` → `read`) inside D1.

Section 10

Payments — Cashfree + Wallet Credits

01
Create Order
`POST /api/payments/create-order` creates order mapping inside D1 and returns Cashfree session ID.
02
Checkout
Cashfree gateway handles checkout in the browser via dynamic UPI/Card forms.
03
Verification
`POST /api/payments/verify` performs order check. If confirmed, it uses D1 batch updates to credit the balance, insert ledger logs, and close the order atomically.
Section 11

Key Design Decisions

DecisionWhatWhy
Client-side Generation Browser renders and exports PDFs Zero server load / memory leaks. Scaled out to user's individual Google quota.
Presigned R2 URLs Direct browser-to-R2 upload Worker never parses PDF buffers in memory, saving CPU and bandwidth.
Edge-Native Hono API Hono deployed on Cloudflare Workers Low latency routing globally, zero-cold start time, free scaling tiers.
Cloudflare D1 Database SQLite DB at the edge Removes legacy PostgreSQL connection overhead and reduces data latency.
Section 12

Interview Q&A

Explain your system design in 60 seconds. ▶
It's a serverless certificate generation platform. The frontend is React hosted on Cloudflare Pages. The API backend is written in Hono and deployed to Cloudflare Workers, using Cloudflare D1 as the edge relational database. When users generate certificates, the frontend copies Slides, replaces placeholders, and uploads the PDF directly to Cloudflare R2 using presigned URLs. For sending, client-side loops hit non-blocking worker endpoints that call Zeptomail or WhatsApp Cloud APIs, ensuring low memory footprint and high scalability.
Why did you choose Cloudflare Workers + D1 over monolithic Express + Postgres? ▶
To reduce hosting costs, eliminate server connection limits, and ensure global low latency. Express backends running on single servers require complex scaling rules and database connection pool management. Cloudflare Workers handle routing globally with zero cold starts, and D1 operates SQLite queries with minimal network overhead.
Section 13

Automated Testing & Coverage

Cephlow uses Vitest to run unit and integration tests across the workspace. It enforces an 80% code coverage threshold (for statements, branches, functions, and lines) using the v8 coverage provider to ensure quality as the codebase grows.

The suite follows a mock-first approach: database queries (D1) and external APIs (Gmail, Google Drive, WhatsApp, Cashfree) are replaced with mocks during unit tests. This ensures tests remain independent, quick, and runnable in any server or CI/CD environment without credentials. Hono routes are tested directly using the Hono app.request() utility to simulate server-side execution.
